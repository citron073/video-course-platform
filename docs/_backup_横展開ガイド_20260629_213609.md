## 2. 完成した骨格（再利用できる構成）

```
<project>/
├── CLAUDE.md            プロジェクト作業ルール（鉄則・フロー・DoD・置き場所）
├── Tasks.md             タスク台帳（完了/残のチェックリスト・都度更新）
├── README.md            使い方の入口（セットアップ・スクリプト）
├── docs/                設計と記録
│   ├── README.md            目次
│   ├── 実装プラン.md         要件・設計の正本（要件変更はここを先に直す）
│   ├── 進捗記録.md           やったこと・教訓ログ
│   ├── 未実行タスク.md       残作業＋再開手順
│   ├── 運用手順.md           seed/デプロイ等の手順書
│   ├── テスト手順.md         テストの使い方・追加方法
│   └── 横展開ガイド.md       ← このファイル
├── app/ components/ db/ lib/   実装（lib/=DB非依存の純粋ロジック＝テスト対象）
└── *.config / package.json
```

役割分担の原則:
- **正本3層を混ぜない** … `実装プラン.md`(設計) / `運用手順.md`(操作) / `Tasks.md`(状態)
- **`docs/`=設計と記録、`README.md`=使い方の入口** で分担

## 3. 横展開テンプレ

### 3-1. 壁打ちで先に潰す分岐点（どんなWeb/アプリ案件でも）
1. 課金/認証の有無（無料公開・限定公開・会員制）
2. データの永続化（DBの要否・どこに置くか）
3. 技術スタックとデプロイ先
4. コンテンツ構成（階層の深さ）
5. 将来拡張（今やらないが設計で余地を残すもの）

→ 決まったら **`docs/実装プラン.md` を正本化** してから実装に入る。

### 3-2. CLAUDE.md に必ず入れるルールの骨子
- 🔴 **テストの鉄則**: テストを通すためにテスト仕様（期待値/ケース）を書き換えない。落ちたら実装を疑う。
- 🔵 **作業フロー**: 着手前に該当 `docs/` を読む → 完了後に `Tasks.md` を更新。
- **DoD**: docs確認 → build成功 → test green → Tasks.md更新 → 進捗記録に1行 → 秘密非コミット。
- **置き場所固定**: 勝手に新場所を作らない。純粋ロジックは `lib/` に集約。

### 3-3. テストしやすくする設計パターン
- **純粋ロジックを `lib/` に切り出す**（DB・env・I/Oに触らない）。
- DBアクセス層はその純粋関数を呼ぶ薄い層にする。
- テストは純粋関数だけを対象に `*.test.ts`（env/DB不要で高速・安定）。
- 境界値（先頭・末尾・空・不在）を必ず入れる。

### 3-4. ドキュメント雛形（新規プロジェクトにコピーして使う）
`docs/` に同名ファイルを置く: `README.md` / `実装プラン.md` / `進捗記録.md` / `未実行タスク.md` / `運用手順.md` / `テスト手順.md`。
ルートに `CLAUDE.md` / `Tasks.md` / `README.md`。**ファイル名は全案件で統一**（迷わない・横断で探せる）。

## 4. 新プロジェクト立ち上げチェックリスト

- [ ] 壁打ちで分岐点（3-1）を確定

- [ ] 純粋ロジックにユニットテスト（Vitest）
- [ ] `CLAUDE.md`（鉄則・フロー・DoD）を設置
- [ ] `Tasks.md`・`docs/` 一式を整備
- [ ] 以後、1作業完了ごとに `Tasks.md` と `docs/進捗記録.md` を更新

## 5. つまずき集（同じ轍を踏まない）

| 症状 | 原因 | 対処 |
|---|---|---|
| `create-next-app .` が失敗 | ディレクトリ名が非ASCII（日本語）でパッケージ名検証に落ちる | 手動スキャフォールド（`package.json` 自作 + `npm install @latest`） |
| seed が `env未設定` で落ちる | ESMのimportホイスティングで dotenv より先にDB層が評価される | DB層を関数内で動的import（`await import("./index")`） |
| サイドバーの現在地が光らない | App Routerのレイアウトは子セグメントのparamを取れない | クライアント化して `usePathname` で判定 |
