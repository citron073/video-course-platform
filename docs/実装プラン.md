# 実装プラン（確定要件・設計の正本）

> 2026-06-28 の壁打ちで確定。要件が変わったらこのファイルを先に更新する。

## なぜ作るか（背景）

YouTube限定公開動画を埋め込んだ、1講座・複数レッスン構成の動画講座サイトを新規構築する。
受講者がレッスンを順番に視聴できる「Udemy風サイドバー型」UIを提供する。

## 確定した要件

| 項目 | 決定 |
|---|---|
| 公開形態 | 無料・限定公開（認証なし／URLを知る人だけがアクセス） |
| 動画 | YouTube限定公開（unlisted）動画を埋め込み |
| 構成 | 1講座 × セクション（章）× レッスン の3階層 |
| UI | サイドバー型（左=セクション/レッスン一覧、右=動画プレーヤー） |
| 進捗管理 | v1では作らない。後付けできるDB設計・コンポーネント構造にする |
| 技術 | Next.js（App Router）／ Turso（libSQL）／ Drizzle ORM |
| デプロイ | Vercel |
| データ管理 | レッスンは Drizzle の seed スクリプトで Turso に投入 |

## アーキテクチャ

```
講座用/
├── app/
│   ├── layout.tsx              ルートレイアウト
│   ├── page.tsx                講座トップ（概要 + 受講をはじめる→最初のレッスン）
│   ├── globals.css
│   └── learn/
│       ├── layout.tsx          サイドバー型レイアウト
│       └── [lessonId]/page.tsx 動画プレーヤー + レッスン情報 + 前/次ナビ
├── components/
│   ├── YouTubeEmbed.tsx        16:9レスポンシブ + nocookie埋め込み
│   ├── LessonSidebar.tsx       セクション別レッスン一覧・現在地ハイライト
│   └── LessonNav.tsx           前/次レッスン
├── db/
│   ├── schema.ts               courses / sections / lessons
│   ├── index.ts                libSQLクライアント + drizzle
│   ├── queries.ts              講座・レッスン取得（Server Componentから呼ぶ）
│   └── seed.ts                 初期データ投入
├── lib/config.ts               COURSE_SLUG（講座スラッグの単一情報源）
└── drizzle.config.ts
```

## DBスキーマ

3テーブル。整数 `order` 列で並び順を管理。

- **courses**: `id`, `slug`(unique), `title`, `description`, `createdAt`
- **sections**: `id`, `courseId`(fk), `title`, `order`
- **lessons**: `id`, `sectionId`(fk), `title`, `youtubeId`, `description`, `durationSeconds`, `order`

外部キーは cascade 削除。seed は冪等（全削除→再投入）。

## 設計上の判断（なぜこうしたか）

- **レッスンデータはDB（Turso）管理**：将来の進捗管理と自然につながるため。設定ファイル直書きにはしない。
- **サイドバーのハイライトはクライアント側 `usePathname`**：App Router のレイアウトは子セグメント（`[lessonId]`）の param を受け取れないため、`LessonSidebar` を client component にして現在地を判定。
- **YouTube は素のiframe（nocookie）**：v1は外部ライブラリ不要。将来 `lite-youtube` 等での遅延読込に差し替え可能。
- **認証なし**：限定公開のため。`robots: noindex` で検索インデックスも抑止。

## 進捗管理の後付け方針（v1では未実装）

`db/schema.ts` 末尾コメントの `progress` テーブルを追加すれば、レッスン側を変更せず拡張できる。

```
progress(id, lessonId, clientId, completed, lastPositionSeconds, updatedAt)
```

認証がないため、当面は匿名 `clientId`（ブラウザ生成→localStorage保持）で進捗を識別する想定。

## 検証方法（Definition of Done）

1. `npm run db:push` → `npm run seed` でデータ投入
2. `npm run dev` で起動し以下を確認
   - `/` に講座トップ、「受講をはじめる」→最初のレッスンへ
   - サイドバーがセクション別に並び、現在レッスンがハイライト
   - 動画が16:9で再生、前/次ナビで移動、端では片側が消える
   - 存在しないIDは404
3. `npm run build` が成功し、コンソールエラーが出ない

# 【ゼミ練習版】EP31〜34：認証・権限の追加（2026-07-20 追記）

> このディレクトリ（`講座用_ゼミ練習/`）は DMM生成AI CAMP「Claude Code」EP31〜34 の練習用コピー。
> 本番の `講座用/` とは別物であり、**本番の「認証なし」設計は変更しない**。

## なぜ要件を変えるか

EP31〜34 は「見た目だけのたたき台に、記憶（DB）・本人確認（ログイン）・権限（管理者）を通す」回。
本番の v1 要件は「認証なし・限定公開」だが、練習版では以下を**意図的に追加**する。

## 変更後の確定要件（練習版のみ）

| 項目 | v1（本番） | 練習版（EP31-34） |
|---|---|---|
| 認証 | なし | **あり**（Better Auth：メール/パスワード + Google OAuth） |
| 権限 | なし | **role**（`admin`=講座を投稿・編集／`user`=受講のみ） |
| 講座の入力 | `db/seed.ts` 直書き | **管理画面から作成**（`/admin`） |
| 公開制御 | なし | **published フラグ**（下書き／公開） |
| DB | ローカル file:./local.db | 同左（Turso 本契約は鍵取得後に切替） |

## 追加スキーマ

Better Auth が要求する固定名テーブル4つ（**名前がズレると動かない**）＋ 既存テーブルの拡張。

- **user**: `id`, `name`, `email`(unique), `emailVerified`, `image`, **`role`**(既定 `user`), `createdAt`, `updatedAt`
- **account**: どのログイン手段か（Google / メール+パスワード）
- **session**: 今ログイン中かどうか
- **verification**: メール確認などの検証用
- **courses**（拡張）: `published`(既定 false), `createdBy`(→user.id), `thumbnailUrl`(EP35で使用)

## 鍵の扱い（重要・人間の手作業）

APIトークン・OAuthシークレットは**AIに渡さない**。以下は必ず本人が実施する。

1. Turso：アカウント作成 → Platform API Token 発行 → `mcp.json` へ（`mcp.json` は `.gitignore` 済）
2. GCP：OAuth 2.0 クライアントID作成 → クライアントID／シークレットを `.env.local` へ
3. リダイレクトURI は `http://localhost:3000/api/auth/callback/google`（**末尾まで完全一致**）
4. 万一 push したら **即 Revoke**（無効化）して再発行

`GOOGLE_CLIENT_ID` / `GOOGLE_CLIENT_SECRET` が未設定の間は、
`lib/auth.ts` が Google プロバイダを**自動で無効化**する（鍵なしでもビルド・起動できる設計）。

## 検証方法（練習版 DoD）

1. `npm run db:push` でテーブル作成（auth 4テーブルを含む）
2. `npm run dev` → `/login` でメール登録 → `user` テーブルに保存されることを確認
3. `role` を `admin` に変更 → `/admin` に入れることを確認
4. `/admin/courses/new` から講座を1つ作成 → 「公開する」で `/` の一覧に出る
5. `npm run build` 成功・`npm test` green

# 【ゼミ練習版】EP35：講座サムネイル画像（Vercel Blob）（2026-07-21 追記）

> 目的：講座に画像を持たせる。置き場は Vercel Blob。**Google認証（EP33）と同じ「鍵があるときだけ有効化」パターン**で、`BLOB_READ_WRITE_TOKEN` 待ちでもビルド・起動できるようにする。

## 設計

- **鍵の有無で有効化**: `lib/blob.ts` の `isBlobEnabled = Boolean(process.env.BLOB_READ_WRITE_TOKEN)`。トークン未設定なら `uploadThumbnail()` は明確なエラーを投げる。鍵は Vercel ダッシュボード → Storage → Blob で発行し `.env.local`（本番は Vercel 環境変数）へ。**AIには渡さない**。
- **純粋ロジックの分離**: ファイル検証は `lib/blob-validate.ts`（DB/env/@vercel/blob 非依存）に切り出しテスト対象にする。`isAllowedImageType`（png/jpeg/webp/gif）／`isWithinSizeLimit`（既定5MB・境界=上限ちょうどはOK）／両者をまとめた `validateThumbnail({type,size})`。テストは `lib/blob-validate.test.ts`（Vitest・15ケース）。
- **アップロード**: `uploadThumbnail(file)` が `@vercel/blob` の `put()` で `access:"public"` + `addRandomSuffix` 保存し公開URLを返す。既存 `courses.thumbnailUrl`（`thumbnail_url`）に保存。
- **フォーム**: `NewCourseForm` に `<input type="file" name="thumbnail" accept="image/*">`。`isBlobEnabled` は Server Component（`page.tsx`）で読み、`blobEnabled` prop で Client に渡す（クライアントで `process.env` を直接読まない）。false のときは input を `disabled` + 案内文。
- **Server Action**: `createCourse` は権限チェック（`canAccessAdmin`）を維持したまま、Blob有効かつファイルありのとき `validateThumbnail`→`uploadThumbnail`。検証NGは既存 `ActionState.errors` で返す。
- **一覧表示**: `app/page.tsx` は `thumbnailUrl` があれば素の `<img>`（外部Blob URLのため next/image不使用）、無ければ「No Image」プレースホルダ（`aspect-video`）。

## 検証（EP35 DoD）

1. `npm run build` 成功・`npm test` green（既存33＋新規15＝48）。
2. トークン未設定でもビルド・起動でき、フォームは input無効＋案内、一覧は No Image で動く。
3. 本番 `講座用/` は不変。`.env.local` の既存値は不変。
