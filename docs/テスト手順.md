# テスト手順（ユニットテスト）

> ひとまずユニットテストのみ。E2E/結合テストは将来必要になったら追加する。

## 使うもの

- **Vitest**（テストランナー）
- 対象は **純粋ロジック**（DB・envに依存しない関数）。`lib/lessons.ts` に集約してある。

## なぜ `lib/lessons.ts` をテストするか

DBアクセス関数（`db/queries.ts`）は `db/index.ts` を import し、env（`TURSO_DATABASE_URL`）が無いと起動時に落ちる。
そこで「並べ替え・グループ化・前後判定」といった**判断ロジックだけを `lib/lessons.ts` に切り出し**、
env/DBなしで高速・安定にテストできるようにしている。`db/queries.ts` はこの純粋関数を呼ぶ薄い層。

## コマンド

```bash
npm test          # 1回実行（CI向け・vitest run）
npm run test:watch # 変更を監視して再実行（開発中）
```

## テスト対象（`lib/lessons.test.ts`）

| 関数 | 確認している挙動 |
|---|---|
| `groupLessonsBySections` | レッスンが正しいセクションに入る／順序維持／空セクションは `[]`／レッスン0件 |
| `flattenLessons` | セクション順→レッスン順でフラット化 |
| `findAdjacentLessons` | 中間=前後あり／先頭=prev null／末尾=next null／1件=両方null／不在ID=両方null／空配列=両方null |

現在 **11 ケース**。すべて green であること。

## テストの書き方（追加するとき）

1. テスト対象は**純粋関数**にする。DBが要るロジックは、まず純粋部分を `lib/` に切り出してからテストする。
2. 入力は `lib/lessons.test.ts` のファクトリ（`makeSection` / `makeLesson`）のように、テスト内で組み立てる。
3. 1ケース1観点。境界値（先頭・末尾・空・不在）を必ず入れる。
4. ファイル名は `*.test.ts`。`lib/` 配下に対象と並べて置く。

## 重要ルール（CLAUDE.md と同義）

**テストを通すためにテスト（期待値・ケース）を書き換えてはいけない。**
テストが落ちたら、まず実装側のバグを疑う。仕様が本当に変わった場合のみ、
**理由を `docs/進捗記録.md` に1行残してから**テストを更新する。
詳細は プロジェクト直下の [`../CLAUDE.md`](../CLAUDE.md) を参照。
