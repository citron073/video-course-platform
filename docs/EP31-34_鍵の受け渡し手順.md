# EP31〜34：鍵の受け渡し手順（たにさん本人の作業）

> コード側はすべて実装済み。ここに書いた**鍵の取得だけは人間の手作業**（AIには渡さない）。
> 資料の原則：`.gitignore` と `.env.local` で守る → GitHubに絶対上げない → 漏れたら即 Revoke。

## A. Turso（EP31・EP32）— データの倉庫

1. https://turso.tech → 「Get Started」→ GitHub か Google で認証
2. ユーザーネームを決める（**これが後で「組織名(organization)」になる**）
3. **APIトークンを発行する（2026年時点の現行仕様）**

   > ⚠️ **講義資料の「右上アバター → Settings → Platform API Token」は古い導線**。
   > Turso は **クロス組織（cross-organization）トークンを廃止**したため、そのページを開くと
   > 「組織間APIトークンはありません／単一の組織にスコープされたトークンを作成するには、
   > その組織内の設定 → API トークン に進んでください」と表示される。**これは正常**。

   正しい導線は「**組織を開いてから**」:
   ```
   ダッシュボード → 組織を選択 → その組織の Settings（設定）→ API Tokens → Create Token
   ```

   UIで迷う場合は CLI が確実（UI変更の影響を受けない）:
   ```bash
   brew install tursodatabase/tap/turso   # 未インストールなら
   turso auth login
   turso auth api-tokens mint claude-code --org <あなたの組織名>
   ```

   - ⚠️ **トークンは発行直後の一度しか表示されない**。その場でコピーする
   - この「組織名」がそのまま `TURSO_ORGANIZATION` に入れる値になる
4. プロジェクト直下（`package.json` と同じ階層）で `cp mcp.json.example mcp.json` して、以下を記入

   | 環境変数 | 中身 | 取得場所 |
   |---|---|---|
   | `TURSO_API_TOKEN` | AIがあなたの代わりにTursoを操作する**合鍵**（超重要・秘密） | 3で発行した値 |
   | `TURSO_ORGANIZATION` | あなたの**組織名**（アカウントの識別名） | `app.turso.tech/<ここ>` の部分。個人ならサインアップ時のユーザー名 |
   | `TURSO_DEFAULT_DATABASE` | 既定のデータベース名（任意） | **今回は設定不要**（空のままでOK） |

   ※ MCPサーバーのパッケージは `mcp-turso-cloud`（`npx -y mcp-turso-cloud` で起動）。

5. Claude Code を再起動して `/mcp` → `mcp-turso-cloud` が **Connected** になればOK
   - `No MCP servers configured` と出たら：`mcp.json` が `package.json` と同じ階層にあるか確認 → Ctrl+C ×2 で終了 → `claude --dangerously-skip-permissions` で再起動 → もう一度 `/mcp`
   - サンプルの `${TURSO_API_TOKEN}` のような**変数のまま**になっていないか確認（実際の値に置換する）
6. Turso ダッシュボードで **Create group**（ロケーションは**東京**）
   - 「group が存在しません」エラーはこれで解消。作ったグループ名をAIに伝える

**本番DBに切り替えるとき**は `.env.local` を書き換える：
```
TURSO_DATABASE_URL=libsql://<db名>-<組織名>.turso.io
TURSO_AUTH_TOKEN=<DB接続トークン>
```
→ `npm run db:push` でテーブルを本番側にも作る。

## B. Google Cloud Platform（EP33）— ログインの許可証

1. https://console.cloud.google.com → 新しいプロジェクト（例: `Video Platform` / 組織なしでOK）
2. **OAuth同意画面**：アプリ名・サポートメール（自分）・対象は必ず **外部（External）**
   - 「internal でログインできない」場合はここが原因。テスト中は自分のメールをテストユーザーに追加
3. **認証情報を作成 → OAuth 2.0 クライアントID → 種類は「ウェブアプリケーション」**
4. **承認済みのリダイレクトURI** に下記を**完全一致**で登録（末尾スラッシュも含めて一致必須）
   ```
   http://localhost:3000/api/auth/callback/google
   ```
5. **クライアントID** と **クライアントシークレット** をコピー
   - シークレットが出ないときは、該当クライアントを開き直す
6. `.env.local` の該当2行のコメントを外して貼る
   ```
   GOOGLE_CLIENT_ID=<クライアントID>
   GOOGLE_CLIENT_SECRET=<クライアントシークレット>
   ```
7. `npm run dev` を**再起動**（環境変数は起動時に読まれる）
   → `/login` に「Googleで続行」ボタンが**自動で出現**する

## C. 自分を管理者にする（EP34）

Googleログイン後、ターミナルで：
```
npm run make-admin -- <自分のメールアドレス>
```
または Claude Code に MCP 経由で頼む：
```
講座を投稿する管理者は <自分のメール> です。
MCP 経由で、このユーザーの role を admin に変更してください。
```
→ **いったんログアウト → 再ログイン**でヘッダーに「管理画面」が出る。

## D. 漏らしてしまったときの復旧

1. **まず Turso / GCP 側でそのトークンを Revoke（無効化）** ← 最優先
2. 新しいトークンを発行し直す
3. そのうえで GitHub の履歴からファイルを削除

## 早見表（つまずき対応）

| 症状 | 対処 |
|---|---|
| `/mcp` で No MCP servers configured | `mcp.json` が `package.json` と同階層か確認 → Ctrl+C ×2 → 再起動 → `/mcp` |
| APIトークンの発行画面が見つからない／「組織間APIトークンはありません」 | クロス組織トークンは**廃止済み**。ダッシュボード → **組織を選択** → その組織の Settings → API Tokens → Create Token。またはCLI `turso auth api-tokens mint <名前> --org <組織名>` |
| 「group が存在しません」 | Turso で Create group（東京）→ グループ名をAIに伝える |
| `redirect_uri_mismatch` | GCPのリダイレクトURIを**完全一致**で登録（`/api/auth/callback/google`） |
| 「Googleで続行」が出ない | `.env.local` の2つの鍵を確認 → `npm run dev` 再起動 |
| ログイン済みなのに管理画面リンクが出ない | role がまだ user → `npm run make-admin` → ログアウト→再ログイン |
| 新規登録で500エラー | `.env.local` の `TURSO_DATABASE_URL` / `BETTER_AUTH_SECRET` を確認 → 再起動 |
| 「パスワードが弱い」 | 8文字以上（大小英字＋数字を混ぜる。例 `Aa12345678`） |
